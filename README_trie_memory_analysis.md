# Investigation: Why the Trie-Based Stat Cache Takes More Memory Than the Map-Based Cache

The `statCacheTrie` was implemented to efficiently support prefix-based cache eviction (`EraseEntriesWithGivenPrefix`) and theoretically save memory by deduplicating long shared file path prefixes. However, benchmark tests (e.g., `TestTrieVsMapMemoryUsageScenarios/HighSharingSharedPrefix`) demonstrated that the Radix Trie actually consumed roughly **15% more memory** than the standard Map-based `lru.Cache`.

This document details the root causes of the memory overhead, the structural constraints, and the fundamental limitations of Go's string memory model that prevent the Trie from using less memory than a flat Map.

## 1. Initial Memory Overheads

There are two major structural memory costs in the Trie that the Map avoids:

### A. Eager Map Allocation
Every `trieNode` unconditionally allocates a `map[byte]*trieNode` for its `children`. In Go, a map has a significant baseline memory footprint (~48 bytes for the `hmap` struct plus bucket allocations).
- **Impact:** For a Trie containing 10,000 leaf nodes (representing files), 10,000 empty maps are allocated. This wastes hundreds of kilobytes of heap memory for nodes that have zero children.

### B. Redundant Heap Allocations (`*entry` vs `entry`)
The `statCacheTrie` allocates its cache entries on the heap (`val *entry`) for every cached object. Go's memory allocator adds overhead (typically 16 bytes) per heap object. The Map-based `lru.Cache` avoids this by storing the `entry` struct by value inside its doubly-linked list nodes.
- **Impact:** 10,000 distinct heap allocations for `entry` objects add significant memory overhead per entry compared to the Map.

## 2. The Fundamental Limitation: Go's String Memory Model

Even if the Trie's structural overheads were optimized out, the Trie still cannot consume less memory than the Map for this specific use case. This remaining difference is entirely due to the inherent structural requirements of a Radix Trie combined with Go's immutable, contiguous string allocation.

### The Deduplication Fallacy
The primary theoretical advantage of a Radix Trie is that it deduplicates shared string prefixes. For example, 10,000 files named `dir/.../shared/prefix/file_0` to `file_9999` share a long common prefix. The Trie stores this 70+ character prefix only once in an internal node, while the leaf nodes only store the differing suffixes (`0`, `1`, etc.).

However, this deduplication yields **zero bytes** of actual memory savings in Go for the `statCacheTrie`.

1. **Strings Must Be Contiguous:** In Go, a `string` is a pointer to a contiguous read-only byte array. Two distinct strings (e.g., `file_0` and `file_1`) cannot safely share the same underlying memory for their prefix while having different suffixes. They must exist as two entirely separate, full-length allocations in memory.
2. **The Cache Holds Full Strings:** The `statCacheTrie` stores `gcs.MinObject` structs. The `MinObject.Name` field is a `string`. Because this field holds the full, concatenated file path, the garbage collector cannot free the underlying contiguous byte arrays for any of the 10,000 full strings generated by the test.
3. **The Map Shares References:** When the Map (`lru.Cache`) stores these 10,000 strings as keys, it does not duplicate the string data. It only allocates a 16-byte string header (pointer + length) that references the *exact same byte array* held by the `MinObject.Name`.

**Conclusion:** The Map pays zero memory penalty for long shared prefixes because it merely holds references to strings that the cache must keep alive anyway. The Trie's deduplication of string headers saves no actual byte-array memory.

### The Structural Overhead of the Radix Trie
While the Trie saves no string memory, it still incurs the structural cost of branching.

For 10,000 sequentially numbered files (`file_0` to `file_9999`), the Radix Trie automatically identifies the shared prefix `file_` and then splits on the differing digits. This inherently requires the Trie to create intermediate branching nodes for the digits `0` through `9`.
- **Impact:** To represent 10,000 sequential files, the Radix Trie must allocate roughly **1,110 internal branching nodes**.
- **Cost:** Each internal node costs approximately ~56 bytes. 1,110 nodes * 56 bytes = ~62 KB.

This ~62 KB structural overhead exactly accounts for the fundamental ~1.5% memory difference between an optimally compressed Trie and the completely flat Map (which requires zero internal branching nodes).

## 3. Why Changing `gcs.MinObject.Name` is Infeasible
To actually deduplicate the strings and allow the Trie to save memory, `gcs.MinObject.Name` would have to be changed from a Go `string` to a more memory-efficient, deduplicable representation such as `[]rune`, `[]byte`, or a custom segmented `Path` struct.

However, implementing this refactor across the GCSFuse codebase presents severe architectural, performance, and maintenance issues:

1. **Incompatibility with External APIs:** `gcs.MinObject` directly represents an object stored in Google Cloud Storage. The underlying Google Cloud SDKs natively consume and return Go `string`s for all object names. Every interaction with GCS would require converting our custom representation back into a standard Go `string`.
2. **Severe Runtime Performance Degradation (GC Churn):** Converting a `[]rune` back into a `string` forces the Go runtime to allocate a brand-new contiguous byte array on the heap. GCSFuse relies heavily on object names for logging, metrics, internal map lookups, and OS-level file system calls. Continuously allocating short-lived strings on the heap thousands of times per second would trigger massive Garbage Collection (GC) churn, drastically reducing the overall throughput and increasing latency and CPU usage of the file system.
3. **Massive Codebase Scope:** `gcs.MinObject.Name` is accessed directly as a `string` in over 60 distinct non-test files across the project. Refactoring this field requires rewriting thousands of lines of code spanning the caching layer, core FUSE filesystem operations, GCS interaction layer, and downloading/prefetching logic.
4. **Loss of Map Key Efficiency:** If `Name` were changed to a slice type, it could no longer be used directly as a map key, because slices are unhashable in Go. All internal indices, lookup tables, and sets that currently key by object name would have to be rewritten.

## Summary
It is theoretically impossible for the `statCacheTrie` to use less memory than the Map implementation when caching `gcs.MinObject`s because:
1. The full file paths must remain resident in memory as contiguous Go strings (`MinObject.Name`).
2. The Map simply points to these existing strings without duplicating their backing arrays.
3. The Radix Trie must allocate additional internal branching nodes (structural overhead) to build its tree, resulting in slightly higher total memory allocation.

The test `TestTrieVsMapMemoryUsageScenarios/HighSharingSharedPrefix` assumes that prefix deduplication will yield absolute memory savings, which is incorrect in Go's memory model for this specific cached struct.
